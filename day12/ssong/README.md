# Day 12 - CPU 스케쥴링, 스케쥴러의 종류

## 프로세스 스케쥴링이란?

OS가 Ready 큐에 있는 프로세스 중 어떤걸 다음 실행으로 선택할지 정하는 것이다. CPU 스케쥴링, Job 스케쥴링이라고도 표현한다.

여러 스케쥴링 기준과 방법이 있지만, 모든 기준을 동시에 만족시키는 최적화된 방법은 없다. 예를 들어, 반응 속도가 빠르면 → 컨텐스트 스위칭이 더 자주 발생해 오버헤드가 증가하고 → 시스템 처리량이 줄어든다.

### 스케쥴링 기준

1. CPU 사용량
2. 처리량
3. 반환 시간(Turnaround time) : 작업이 모두 끝날 때까지 걸리는 시간
4. 대기시간 : 큐에서 대기하는 시간
5. 응답 시간 : 요청에 대한 첫 응답이 오는 시간
6. 공정성 : 사용자들 간 CPU를 공정하게 공유하는지

### 스케쥴링의 대전제: 선점 vs 비선점

- 선점형(Preemptive) : OS가 실행 중인 프로세스의 CPU를 강제로 빼앗아 다른 프로세스에게 줄 수 있음
- 비선점형(Non-preemptive) : 한 프로세스가 CPU를 잡으면, 작업이 끝나거나 스스로 대기(Wait) 상태로 가기 전까지는 CPU를 빼앗지 않음

## 스케쥴러의 종류

#### FIFO(First-In, First-Out)

- 방식: 비선점형
- 원리: 먼저 도착한 프로세스가 먼저 CPU를 사용
- 단점: 앞에 실행 시간이 엄청 긴 프로세스로 인해 뒤에 있는 짧은 프로세스들이 하염없이 기다려서 평균 대기 시간이 길어진다는 **콘보이 효과(Convoy Effect)**가 있음

#### SJF (Shortest-Job-First)

- 방식: 비선점형, 선점형 둘 다 가능(선점형 SJF는 보통 SRT라고 부름)
- 원리: 실행 시간이 가장 짧은 프로세스에게 CPU를 먼저 줌
- 장점: 평균 대기 시간을 최소화
- 단점: 실행 시간이 긴 프로세스들이 짧은 프로세스 뒤로 계속 밀려 영원이 실행되지 않는 기아 현상(Stravation)이 있음.

#### Round Robin (RR)

- 방식: 선점형(현대 시분할 시스템의 모태)
- 원리: 각 프로세스는 동일한 크기의 시간(Time slice)을 할당 받음. 자기 차례에 할당된 시간이 지나면 CPU를 빼앗기고 큐의 맨 뒤로 이동함
- Time Slice가 길면 FIFO와 동일해지고, 짧으면 Context Switching 오버헤드가 커짐

#### MLQ (Multi Level Queue)

- 방식: 선점형
- 원리: 우선순위에 따라 큐를 구분지어 실행. 각 큐는 독자적인 스케쥴링 알고리즘을 통해 스케쥴링 수행
- 단점: 큐 사이에 강한 우선순위가 있어, 하위 큐의 기아 현상이 발생할 수 있음

#### MLFQ (Multi Level Feedback Queue)

- 방식: 선점형
- 원리: MLQ와 비슷한데, 프로세스가 큐 사이를 이동할 수 있음. 동작 원리는 다음과 같다.
  1. 처음 들어온 프로세스는 가장 높은 우선순위 큐에 들어간다
  2. 주어진 시간(Time slice) 내에 작업을 못끝내면 한 단계 아래 우선순위 큐로 내려간다
  3. 반대로 I/O를 많이해서 CPU를 양보하는 프로세스는 높은 우선순위 큐에 머무른다
- 장점: CPU 중심 프로세스는 하위 큐로 내려가고, 반응성이 중요한 프로세스는 상위 큐에 있어 효율적이다.
- 단점: 대화형 프로세스가 너무 많으면 작업이 긴 프로세스들이 CPU를 얻지 못하는 기아현상이 발생한다. 또한, 하위 큐에 내려가면 영원히 실행되지 못할 수도 있다.
  이를 해결하기 위해 Priority Boost를 추가해야 한다. 이는 일정 시간이 지나면 모든 작업들을 최상위 큐로 올려주는 방법이다.
