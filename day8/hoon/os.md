# day 8-1 운영체제란?
## 1. 운영체제란?
```text
컴퓨터의 하드웨어 자원을 관리하고, 응용 프로그램이 하드웨어를 편리하고 안전하게 사용할 수 있도록 서비스를 제공하는 시스템 소프트웨어
```
```text
[운영체제의 구조]

사용자 -> 응용 프로그램 -> 운영체제 -> CPU • 메모리 • 저장장치 • 입출력 장치

```
## 운영체제의 두 가지 관점

### 자원 관리자
- CPU, 메모리, 저장장치, 입출력 장치를 여러 프로그램에 배분

### 추상화 제공자
- 응용 프로그램이 복잡한 하드웨어를 직접 조작하지 않아도 되도록 간단한 인터페이스를 제공 (a.k.a 리모컨)

## 2. 운영체제의 목적

### 편리성
- 사용자와 응용 프로그램이 하드웨어를 쉽게 사용할 수 있도록 인터페이스 제공

### 효율성
- CPU, 메모리, 저장장치 등은 유한하다. 이를 효율적으로 사용

### 보호 / 공정성
- 프로세스가 다른 프로세스의 메모리를 침범하지 못하도록 보호
- 특정 프로세스가 자원을 독점하지 않도록 관리

```text
운영체제는 프로세스 관리, 메모리 관리, 파일 시스템 관리, 입출력 관리 등 다양한 기능을 담당한다.
```
## 3. 커널 / 사용자 영역

### 커널의 역할
- 높은 권한을 가지고 CPU, 메모리, 프로세스, 파일 시스템, 입출력 장치 등 컴퓨터의 핵심 자원을 관리함

### 사용자 영역
- 일반 응용 프로그램이 실행되는 영역
- 응용 프로그램은 하드웨어나 커널 메모리에 직접 접근할 수 없음, 운영체제가 제공하는 인터페이스를 통해 필요한 작업을 요청

Windows에서도 각 사용자 모드 응용 프로그램은 독립적인 가상 주소 공간을 가짐, 커널 공간에는 직접 접근 x
```text

사용자 모드
- 웹 브라우저
- 문서 편집기
- 게임
- 일반 응용 프로그램

──────── 시스템 콜 ────────

커널 모드
- CPU 스케줄러
- 메모리 관리자
- 파일 시스템
- 장치 드라이버

```
```text
- 시스템 콜이란?
사용자 프로그램이 운영체제의 기능을 요청하기 위해 사용하는 공식적인 인터페이스
(파일 열기, 파일 읽고 쓰기, 프로세스 생성, 메모리 할당, 네트워크 통신 등) 
```

## 4. 운영체제의 핵심 자원 관리

### 4.1. CPU와 프로세스 관리

`프로그램`: 저장장치에 보관된 명령어와 데이터의 집합
`프로세스`: 실행 중인 프로그램

```text
[운영체제가 관리하는 것]
- 프로세스 생성 / 종료
- CPU 할당
- 프로세스 상태 관리
- 프로세스 간 통신과 동기화
- 교착상태 처리
```

`현대 컴퓨터는 대부분 멀티코어`
하나의 CPU 코어는 한 시점에 하나의 명령 흐름을 실행함.
운영체제는 여러 프로세스와 스레드에 CPU 시간을 빠르게 나누어 할당하며, 여러 코어가 있다면 일부 작업은 실제로 병렬 실행될 수 있음.

프로세스의 관리 정보(CPU 스케줄링 정보, 메모리 관리 정보, 입출력 상태 등)는 `PCB`에 저장됨

[스케줄링 알고리즘]
| 알고리즘        | 기준             | 특징                             |
| ----------- | -------------- | ------------------------------ |
| FCFS        | 먼저 도착한 프로세스    | 단순하지만 긴 작업 때문에 짧은 작업이 기다릴 수 있음 |
| Round Robin | 일정 시간만큼 순환 실행  | 대화형 시스템에 적합                    |
| Priority    | 우선순위가 높은 프로세스  | 낮은 우선순위 프로세스의 기아 가능            |
| SJF         | 실행 시간이 짧은 프로세스 | 평균 대기시간을 줄일 수 있음               |

### 4.2. 메모리 관리
`메모리`: CPU가 실행할 명령어와 데이터를 저장하는 공간.
+ 명령어는 실행되기 전에 메모리에서 CPU로 전달되어야 하므로 `메모리 관리`는 운영체제의 핵심 기능이다.

```text
[운영체제 수행 작업]
- 프로세스에 메모리 할당 / 회수
- 프로세스별 주소 공간 보호
- 가상 주소를 물리 주소로 변환
- 필요한 메모리를 메모리로 불러옴
- 메모리 부족할 경우 교체할 페이지 결정
```
#### 연속 메모리 할당
- `고정 분할`
메모리를 미리 정해진 크기의 여러 영역으로 나누는 방식
    - 한 분할에는 하나의 프로세스가 적재됨
    - 분할의 크기보다 작은 프로세스를 넣을 경우 `내부 단편화` 발생

- `가변 분할`
프로세스 크기에 맞게 메모리 영역을 할당하는 방식
    - 남은 공간이 여러 곳에 작게 흩어질 경우 `외부 단편화` 발생

| 구분     | 낭비되는 공간       |
| ------ | ------------- |
| 내부 단편화 | 할당된 메모리 영역 내부 |
| 외부 단편화 | 할당되지 않은 영역 사이 |


#### 가상 메모리
- 각 프로세스에 독립적인 가상 주소 공간을 제공
-> 필요한 일부만 물리 메모리에 적재하는 방식
- 운영체제와 하드웨어는 `페이지 테이블`을 이용해 가상 주소를 물리 주소로 변환함
```text
프로세스가 사용하는 가상 주소
           ↓
       페이지 테이블
           ↓
물리 메모리의 실제 주소
```
```text
[페이지 vs 프레임]
페이지: 가상 주소 공간을 나눈 일정한 크기의 단위
프레임: 물리 메모리를 나눈 일정한 크기의 단위

페이지와 프레임의 크기는 같음
페이지 테이블은 페이지가 어느 프레임에 있는지 기록
```
+ Windows 문서
프로세스가 사용하는 가상 주소와 RAM의 물리 주소 사이의 변환을 운영체제가 관리하는 변환 테이블과 하드웨어가 처리함

### 4.3 파일 및 저장장치 관리

운영체제는 저장장치의 데이터를 파일과 디렉터리 형태로 관리함
 ```text
- 파일과 디렉터리 생성 및 삭제
- 파일 읽기와 쓰기
- 파일의 이름, 경로, 접근 권한 관리
- 저장장치 공간 할당 및 회수
 ```

 운영체제는 파일이라는 논리적인 단위를 실제 저장장치의 물리적인 블록과 연결함

### 4.4 입출력 장치 관리
 운영체제는 장치 드라이버를 통해 각 장치를 공통된 방식으로 사용할 수 있도록 함
 -> 입출력 장치 관리는 운영체제의 중요한 하위 시스템 중 하나
```text
응용 프로그램
     ↓
운영체제의 입출력 인터페이스
     ↓
장치 드라이버
     ↓
장치 컨트롤러
     ↓
실제 장치
```

## 5. 운영체제의 작업 처리 방식과 시스템 구성

### 일괄 처리
여러 작업을 모아 순서대로 처리함
- 사용자와의 상호작용이 거의 없음
- 처리 결과를 받기까지 시간이 길 수 있다

### 다중 프로그래밍
여러 프로그램을 메모리에 올려놓고, 한 프로그램이 입출력을 기다리는 동안 다른 프로그램을 실행함
-> `CPU 이용률 향상`

### 시분할
CPU 시간을 짧은 단위로 나누어 여러 사용자나 프로세스에 번갈아 할당함
-> `빠른 응답성 / 대화형 사용 환경 제공`

### 다중 처리
하나의 시스템에 CPU / CPU 코어가 여러 개 있어 작업을 실제로 병렬 처리할 수 있는 방식

### 실시간 처리
정해진 시간 제약을 만족해야 하는 방식

| 종류     | 특징                          | 예             |
| ------ | --------------------------- | ------------- |
| 경성 실시간 | 마감 시간을 놓치면 시스템 실패로 간주       | 항공 제어, 원자로 제어 |
| 연성 실시간 | 지연이 품질 저하를 일으키지만 치명적 실패는 아님 | 영상·음성 스트리밍    |


## 6. 인터럽트

- CPU의 현재 실행 흐름을 일시적으로 변경하여 특정 사건을 처리하도록 만드는 메커니즘
- 하드웨어 장치가 CPU의 처리를 요구할 때 주로 사용함
(ex) 키보드 입력, 디스크 입출력 완료, 네트워크 패킷 도착, 타이머 만료

```text
[인터럽트 처리 과정]
1. CPU가 현재 프로그램을 실행한다. 
2. 장치가 인터럽트 요청을 발생시킨다.
3. CPU가 현재 실행 흐름을 중단하고 필요한 실행 문맥을 저장한다.
4. 운영체제가 해당 인터럽트 처리 루틴을 실행한다.
5. 인터럽트 처리가 끝나면 저장한 실행 문맥을 복원한다.
6. 중단했던 프로그램을 다시 실행한다.
```
+ CPU는 프로그램 전체를 저장하는 것이 아니라, 다시 실행하기 위해 필요한 프로그램 카운터와 레지스터 등의 실행 문맥을 저장한다.
+ `인터럽트 처리 루틴`: 인터럽트가 발생했을 때 수행할 작업을 정의한 커널 코드. 인터럽트 핸들러 혹은 ISR이라 불림

## 참고
- PCB (Process Control Block)
    - 운영체제가 각 프로세스를 관리하기 위해 유지하는 커널의 자료구조
    - 프로세스의 상태와 실행에 필요한 정보를 기록해 둔 관리 카드라 생각할 수 있음
    - 프로세스마다 하나씩 생성되며, 운영체제의 커널 영역에서 관리됨
```text
- PCB에 저장되는 주요 정보
  - 프로세스 식별자(PID)
  - 프로세스 상태
  - 프로그램 카운터
  - CPU 레지스터 값
  - CPU 스케줄링 정보
  - 메모리 관리 정보
  - 입출력 및 열린 파일 정보
```

- 시스템 콜과 인터럽트의 차이
| 구분        | 발생 원인                | 예               |
| --------- | -------------------- | --------------- |
| 하드웨어 인터럽트 | 외부 장치                | 키보드 입력, 디스크 완료  |
| 예외        | 현재 명령 실행 중 발생        | 0으로 나누기, 페이지 폴트 |
| 시스템 콜     | 프로그램이 의도적으로 커널 기능 요청 | 파일 읽기, 프로세스 생성  |
