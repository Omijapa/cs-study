# day17-3 Index

## 1. Index란
- 데이터베이스에서 원하는 데이터를 빠르게 찾기 위해 사용되는 별도의 자료구조
- 책에서 원하는 내용을 찾을 때 처음부터 모든 페이지를 읽는 대신, 맨 뒤의 색인을 이용하는 것과 비슷함

```text
인덱스 없음
→ 테이블의 데이터를 처음부터 끝까지 확인

인덱스 있음
→ 인덱스에서 위치를 찾은 후 해당 데이터에 접근
```

(ex) 회원 테이블에서 특정 이메일을 찾는다 가정
```sql
SELECT *
FROM 회원
WHERE 이메일 = 'user@example.com';
```

- 이메일 컬럼에 인덱스가 있다면 인덱스를 통해 해당 데이터의 위치를 빠르게 찾을 수 있음

## 2. Index 생성

```sql
CREATE INDEX idx_회원_이메일
ON 회원(이메일);
```

- `idx_회원_이메일`: 인덱스 이름
- `회원`: 대상 테이블
- `이메일`: 인덱스를 생성할 column

### 인덱스 삭제

```sql
DROP INDEX idx_회원_이메일;
```

## 3. 인덱스 동작 방식

```text
인덱스 키 값 -> 실제 데이터의 위치

(ex)
kim@example.com  → 회원 데이터 위치 3
lee@example.com  → 회원 데이터 위치 8
park@example.com → 회원 데이터 위치 12

```

- DB는 먼저 인덱스에서 값을 검색한 뒤, 인덱스가 가리키는 실제 데이터에 접근함
- 주로 `B-Tree 계열 자료구조`가 사용됨

## 4. B-Tree Index
- 데이터를 정렬된 상태로 저장하는 트리 자료구조
- 트리의 높이를 따라 검색 -> 전체 데이터를 순차적으로 확인하는 것보다 빠름

### 특징
- 검색 속도가 빠름
- 데이터가 정렬되어 있음
- 범위 검색에 유리함
- 대부분의 관계형 데이터베이스에서 기본 인덱스 구조로 사용됨

## 5. 인덱스의 장단점

### 장점
- 검색 속도 향상
- 정렬 성능 향상
- JOIN 성능 향상

### 단점
- 추가 저장 공간 필요
- INSERT 성능 저하
- UPDATE 성능 저하
- DELETE 성능 저하

## 6. 인덱스의 종류

### 단일 인덱스
- 하나의 컬럼으로 구성된 인덱스

```sql
CREATE INDEX idx_name
ON 회원(이름);
```

### 복합 인덱스
- 두 개 이상의 컬럼으로 구성된 인덱스

```sql
CREATE INDEX idx_region_age
ON 회원(지역, 나이);
```

### 고유 인덱스
- 중복 값을 허용하지 않는 인덱스

```sql
CREATE UNIQUE INDEX idx_email
ON 회원(이메일);
```

## 7. 인덱스가 효율적인 경우 / 비효율적인 경우

### 효율적인 경우
- WHERE 조건에 자주 사용되는 컬럼
- JOIN, ORDER BY에 자주 사용되는 컬럼
- 중복 값이 적은 컬럼
- 전체 데이터 중 일부만 조회하는 경우

### 비효율적인 경우
- 데이터가 적은 테이블
- 중복 값이 많은 컬럼
- 데이터 변경이 매우 빈번한 컬럼
- 전체 데이터의 대부분을 조회하는 경우

## 참고
### 면접 답변 예시
```text
인덱스는 데이터 조회 속도를 높이는 별도의 자료구조입니다. 주로 B-Tree 계열을 사용하며, WHERE, JOIN, ORDER BY에 자주 사용되는 컬럼에 생성합니다. 다만 추가 저장 공간이 필요하고, INSERT, UPDATE, DELETE 시 인덱스도 함께 수정해야 하므로 변경 성능이 저하될 수 있습니다. 따라서 조회 빈도와 중복도, 복합 인덱스의 컬럼 순서를 고려해 생성해야 합니다.
```
